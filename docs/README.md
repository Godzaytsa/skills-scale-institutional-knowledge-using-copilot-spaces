# OctoAcme Project Management Documentation

Welcome to the OctoAcme project management process library. These documents provide guidance for running projects with clarity, accountability, and continuous improvement.

## Overview

OctoAcme follows a structured five-phase project lifecycle: **Initiation → Planning → Execution → Release → Close & Retrospective**. We emphasize iterative delivery, clear role ownership, data-informed decisions, and psychological safety to enable teams to deliver customer value efficiently.

### Key Principles
- **Customer-first**: Prioritize customer value and usability
- **Iterative delivery**: Deliver small, testable increments
- **Clear ownership**: Each project has a named Project Manager and Product Lead
- **Data-informed decisions**: Measure impact and iterate based on evidence
- **Psychological safety**: Encourage feedback and learning

## Core Roles

- **Project Managers** coordinate delivery, schedules, risks, and communications
- **Product Managers** define outcomes, prioritize the backlog, and measure success
- **Developers** implement features with quality standards
- **QA/Testing** validates acceptance criteria and quality

## Communication Cadence

- **Daily**: 15-minute standups focusing on progress, blockers, and dependencies
- **Weekly**: PM/PdM alignment sync and delivery team syncs
- **Monthly**: Stakeholder updates
- **Ad-hoc**: Escalations and incident management

## Quality & Execution Standards

Our execution approach includes:
- Small, focused pull requests (≤400 lines when possible)
- Automated testing (unit, integration, end-to-end smoke tests)
- Security scanning in CI/CD pipelines
- Manual QA for feature acceptance
- Weekly risk register reviews
- Structured blocker escalation (Team → PM → Product Lead → Sponsor)

## Retrospectives & Continuous Improvement

After each sprint, release, or milestone, we hold 45–75 minute retrospectives to discuss:
- What went well
- What could improve
- Prioritized action items with clear owners and timelines

Action items are tracked in the project backlog and reviewed in weekly PM syncs, with impact measured to celebrate improvements and drive iterative change.
